# App Audit — Functionality & Structure — 2026-03-22

## Scope

This version of the audit intentionally excludes visual UI, theming, copy, and presentation concerns.
It focuses only on:
- application functionality
- code structure
- state management
- IPC boundaries
- persistence
- runtime behavior
- tooling and maintainability

---

## Executive summary

### Tooling status
- `npm run typecheck` ✅ passes
- `npm run lint` ❌ fails with **436 errors** and **4216 warnings**

### Prioritized summary
- **1 critical issue** around session identity/persistence
- **5 high-severity issues** around architecture, state, typing, and runtime behavior
- **3 medium-severity issues** around consistency of data flow and resource management
- **2 low-severity issues** around repo/build hygiene and test coverage

### Top issues to fix first
1. **Session identity is not canonical** between local app state and Pi SDK state.
2. **Session lifecycle is modeled ambiguously** between drafts, local-only sessions, and persisted Pi-backed sessions.
3. **IPC/event contracts are too loosely typed**, especially across preload and renderer.
4. **Several React state/ref patterns are brittle** and flagged by modern React lint rules.
5. **Concurrent session initialization can duplicate work** in the runner.

### Overall assessment
The app has a solid baseline separation between main, preload, and renderer, but the most important runtime paths — session creation, persistence, event delivery, and renderer state synchronization — still need structural cleanup. The biggest risk is not UI polish; it is that session identity and state flow are not fully unified yet.

## Detailed findings by severity

## Critical issues

### 1. Session identity and persistence are inconsistent
- **Location:** `src/main/services/session-manager.ts`, `src/main/services/pi-runner.ts`
- **Severity:** Critical
- **Category:** Persistence / state model
- **Description:** `createSession()` creates a local app session using `randomUUID()`, but `ensureAgentSession()` later creates a separate Pi SDK session via `SessionManager.create(session.repoPath)`, which has its own id and backing file.
- **Impact:** The system can end up with two identities for what users think is one session. That risks mismatched file tracking, incorrect resume behavior, duplicate records, and hard-to-debug lifecycle bugs after reload or restart.
- **Recommendation:** Make the Pi SDK session the canonical source of truth at creation time. The app should store and expose the SDK-managed id/file immediately instead of creating a separate local identity.
- **Suggested command:** `/normalize`

## High-severity issues

### 2. Session lifecycle is modeled ambiguously
- **Location:** `src/main/services/session-manager.ts`
- **Severity:** High
- **Category:** Architecture / data model
- **Description:** `listSessions()` merges Pi-discovered sessions with local in-memory sessions via `localOnly`, but there is no explicit lifecycle model for how a draft becomes a persisted Pi-backed session.
- **Impact:** The mental model of a session is fuzzy. That increases risk when implementing deletion, resume, import/export, crash recovery, and status transitions.
- **Recommendation:** Define a single explicit session lifecycle. If drafts must exist before persistence, model them clearly and define the exact promotion path into canonical persisted sessions.
- **Suggested command:** `/normalize`

### 3. Main ↔ preload ↔ renderer contracts are too loosely typed
- **Location:** `src/shared/session.ts`, `src/preload/index.ts`, `src/preload/index.d.ts`, `src/renderer/src/lib/sessions.ts`, `src/renderer/src/components/shell/pi-messages.tsx`, `src/renderer/src/components/shell/browser-view.tsx`, `src/renderer/src/components/shell/terminal-view.tsx`
- **Severity:** High
- **Category:** Type safety / architecture
- **Description:** The app relies heavily on `unknown`, `unknown[]`, `any`, and cast-heavy handling for agent messages, streaming events, and global preload APIs. `window.browser` and `window.terminal` are exposed in preload but not declared on `Window`.
- **Impact:** TypeScript cannot protect the most important runtime boundary in the app. Schema drift can silently break message rendering or session behavior.
- **Recommendation:** Introduce shared domain types for agent messages, streaming events, terminal API, and browser API, and use them consistently from main through renderer.
- **Suggested command:** `/normalize`

### 4. React state/ref handling is structurally brittle in key flows
- **Location:** `src/renderer/src/hooks/use-session-state.ts`, `src/renderer/src/routes/__root.tsx`, `src/renderer/src/components/shell/pi-messages.tsx`, `src/renderer/src/components/shell/session-conversation.tsx`
- **Severity:** High
- **Category:** State management / runtime correctness
- **Description:** The code currently includes synchronous state updates inside effects and ref reads/writes during render. Lint flags these directly with `react-hooks/set-state-in-effect` and `react-hooks/refs`.
- **Impact:** This can produce extra renders, subtle StrictMode issues, stale state behavior, and fragile scroll/session synchronization.
- **Recommendation:** Move mutations out of render, prefer derived state where possible, and refactor session/message state transitions into clearer effect or reducer-based flows.
- **Suggested command:** `/normalize`

### 5. Concurrent session initialization can duplicate work
- **Location:** `src/main/services/pi-runner.ts`, `src/renderer/src/components/shell/model-selector.tsx`
- **Severity:** High
- **Category:** Runtime behavior / performance
- **Description:** `ensureAgentSession()` caches created sessions but not in-flight initialization promises. If two calls arrive at once, the same session can be initialized multiple times. `ModelSelector` is one place where parallel calls can trigger this.
- **Impact:** Duplicate subscriptions, redundant initialization, and inconsistent state flow are possible.
- **Recommendation:** Cache pending initialization promises by `sessionId` and resolve all concurrent callers from the same promise.
- **Suggested command:** `/optimize`

### 6. Lint configuration does not reflect ownership boundaries in the codebase
- **Location:** `eslint.config.mjs`, `src/renderer/src/components/ui/*`, `src/renderer/src/components/ai-elements/*`, `src/renderer/src/routeTree.gen.ts`
- **Severity:** High
- **Category:** Maintainability / tooling
- **Description:** Generated, vendored, or scaffold-derived files are being linted with the same strict rules as first-party application logic. This floods the repo with noise.
- **Impact:** The quality gate is not trusted, which means structurally important warnings in app-authored code are easy to miss.
- **Recommendation:** Separate first-party app rules from generated/scaffolded code. Exclude generated files, and add rule overrides for third-party-style component folders.
- **Suggested command:** `/normalize`

## Medium-severity issues

### 7. Message normalization is duplicated inside the renderer view layer
- **Location:** `src/renderer/src/components/shell/pi-messages.tsx`
- **Severity:** Medium
- **Category:** Code structure
- **Description:** The renderer reconstructs the agent message model locally with component-level helper types and cast-based parsing.
- **Impact:** Message parsing logic is tightly coupled to the UI surface and can drift from the runner or SDK over time.
- **Recommendation:** Normalize messages once in a shared adapter/domain layer, and pass renderer components a stable internal shape.
- **Suggested command:** `/extract`

### 8. Session ordering can drift after realtime updates
- **Location:** `src/renderer/src/lib/workspace.ts`
- **Severity:** Medium
- **Category:** Data consistency
- **Description:** Initial sessions are sorted in `listSessions()`, but `upsertSession()` replaces or appends without re-sorting.
- **Impact:** Live-updated sessions can appear in an outdated order, which means the UI no longer reflects actual most-recent activity.
- **Recommendation:** Re-sort after upsert or centralize sorting in one place so ordering is deterministic.
- **Suggested command:** `/normalize`

### 9. Browser and terminal resource lifecycle needs clearer ownership
- **Location:** `src/main/services/browser.ts`, `src/main/services/terminal.ts`, `src/renderer/src/components/shell/terminal-view.tsx`
- **Severity:** Medium
- **Category:** Resource management
- **Description:** Browser views are detached but not disposed per session, and terminal PTYs intentionally persist after unmount.
- **Impact:** Long-running app sessions can accumulate hidden background resources, especially across many projects or tool-tab toggles.
- **Recommendation:** Define explicit retention and disposal rules. Resources that survive unmount should be deliberate and bounded.
- **Suggested command:** `/optimize`

## Low-severity issues

### 10. Build and repo hygiene are inconsistent
- **Location:** `package.json`, `.gitignore`, repository root, `src/renderer/src/renderer/src`
- **Severity:** Low
- **Category:** Project structure / hygiene
- **Description:**
  - `build:mac` does not follow the same shared build path as `build:win` and `build:linux`
  - `tsconfig.node.tsbuildinfo` and `tsconfig.web.tsbuildinfo` are tracked in git
  - there is an odd nested path `src/renderer/src/renderer/src` that appears to be dead layout residue
- **Impact:** These do not break core functionality immediately, but they increase confusion and maintenance cost.
- **Recommendation:** Standardize build scripts, remove tracked generated artifacts, and delete dead directories.
- **Suggested command:** `/normalize`

### 11. No app-level test or CI safety net was found
- **Location:** repo-wide
- **Severity:** Low
- **Category:** Maintainability
- **Description:** No application test suite or CI workflow was found outside dependency directories.
- **Impact:** Refactoring session creation, persistence, and IPC behavior is riskier than it should be.
- **Recommendation:** Add a minimal smoke suite covering session creation, session persistence, IPC contract behavior, and session ordering.
- **Suggested command:** `/harden`

## Patterns and systemic issues

### 1. Canonical ownership is unclear in core data flows
The app has multiple places that can appear to own session truth:
- local in-memory session maps
- Pi session files
- runner-managed agent session instances
- renderer-local message/session state

This is the main architectural problem.

### 2. Boundary typing is weak exactly where runtime risk is highest
The least-typed areas are the ones that most need stability:
- IPC payloads
- streaming events
- message shapes
- preload globals

### 3. Runtime synchronization is more complex than it looks
The renderer currently rehydrates local state from loaders, subscriptions, and streaming events at once. That can work, but only if the transitions are explicit and canonical.

### 4. Tooling noise is hiding structural defects
The repo has real architecture/state problems, but the lint signal is diluted by unrelated generated/scaffolded code issues.

## Positive findings

### What is working well
- Main, preload, and renderer are separated cleanly.
- IPC is organized by domain (`sessions`, `browser`, `terminal`), which is a strong baseline.
- Shared session/project types already exist, so there is a good place to centralize contracts.
- `npm run typecheck` passes, which means the current baseline is stable enough for targeted refactoring.
- The app is modular enough that session, browser, and terminal concerns can be improved independently.

## Recommendations by priority

### Immediate
1. Make Pi SDK session creation the canonical identity/persistence path.
2. Define an explicit session lifecycle model.
3. Strengthen shared typing for IPC, events, and preload globals.
4. Fix React state/ref violations in the session shell.

### Short-term
5. Deduplicate concurrent `ensureAgentSession()` calls.
6. Move message normalization out of the renderer view layer.
7. Re-sort sessions consistently after updates.
8. Clean up lint config so structural defects become visible.

### Medium-term
9. Add explicit disposal/retention rules for browser and terminal resources.
10. Standardize build/dev hygiene and remove dead/generated repo noise.
11. Add smoke tests and CI coverage around core lifecycle paths.
